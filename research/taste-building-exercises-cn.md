> 面向非程序员的产品感觉、UX深度分析与视觉品味训练

## 第一部分：产品感觉 (Product Sense) — 10个练习

---

### 练习 1：RICE 优先级排序实战

**中文名称：** 「功能价值称重器」

**时间估计：** 60 分钟

**格式说明：** 个人书面练习，使用电子表格完成 RICE 评分，对比结果与直觉判断的差异。

#### 具体步骤

**第一步：准备功能清单（10分钟）**

选择一个你每天使用的应用（如微信、抖音、美团），列出 8 个你希望它新增的功能。例如：

| 编号 | 功能描述 |
|------|----------|
| F1 | 微信增加「消息定时发送」功能 |
| F2 | 微信增加「聊天记录云端搜索」 |
| F3 | 微信增加「朋友圈评论表情包」 |
| F4 | 微信增加「群聊投票增强版」 |
| F5 | 微信增加「语音消息倍速播放」 |
| F6 | 微信增加「已读回执可选开关」 |
| F7 | 微信增加「深色模式定时切换」 |
| F8 | 微信增加「聊天背景按联系人设置」 |

**第二步：逐项打分（30分钟）**

为每个功能打分（使用以下标准）：

- **Reach（触达）：** 每季度有多少用户会使用？
  - 10 = 几乎所有用户（>80%）
  - 6 = 大部分用户（40-80%）
  - 3 = 部分用户（10-40%）
  - 1 = 少数用户（<10%）

- **Impact（影响）：** 对用户满意度的影响程度？
  - 3 = 巨大影响（改变使用习惯）
  - 2 = 较大影响（明显提升体验）
  - 1 = 中等影响（有所改善）
  - 0.5 = 轻微影响（锦上添花）

- **Confidence（信心）：** 你对以上判断的把握度？
  - 100% = 有数据支撑（用户调研、竞品已验证）
  - 80% = 较有把握（部分数据 + 经验判断）
  - 50% = 不太确定（主要靠直觉）

- **Effort（工作量）：** 预估需要多少「人月」？
  - 1 = 极简（一人一周）
  - 3 = 中等（一人一月）
  - 6 = 较大（团队一个月）
  - 12 = 巨大（团队一季度）

**第三步：计算 RICE 分数（10分钟）**

公式：`RICE = (Reach x Impact x Confidence) / Effort`

将 8 个功能按 RICE 分数从高到低排序。

**第四步：反思与对比（10分钟）**

回答以下问题：
1. RICE 排序和你直觉中的优先级一致吗？哪些不同？
2. 哪个因素对结果影响最大（Reach 还是 Effort）？
3. 如果你是产品经理，你会选择 RICE 最高的还是你直觉最强的？为什么？

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 8个功能均有合理评分依据，反思部分有独立见解 |
| 良好 | 评分合理但部分缺乏依据，反思较浅 |
| 需改进 | 评分随意，无法解释打分逻辑 |

#### 预期收获

- 理解如何用数据化方式取代「我觉得这个好」的主观判断
- 认识到 Effort 在决策中的杠杆作用
- 建立「价值/成本」思维模型

### 练习 2：Kano 模型功能分类

**中文名称：** 「用户期望分层器」

**时间估计：** 75 分钟

**格式说明：** 模拟用户调研 + 分类练习，需要 3-5 位朋友配合回答问题。

#### 具体步骤

**第一步：选择产品和功能列表（10分钟）**

选择一个大家都用的产品（如外卖App），列出 10 个功能：

```
1. 下单后实时显示骑手位置
2. 一键重复上次订单
3. AI 根据天气推荐菜品
4. 提前30分钟提醒「该点午餐了」
5. 下单时显示餐厅实时排队情况
6. 免费送餐（无起送费）
7. 订单完成后自动生成饮食热量报告
8. 骑手送达后拍照确认
9. 支持多人拼单 AA 自动分账
10. 食物过敏原标注
```

**第二步：设计 Kano 问卷（10分钟）**

对每个功能，设计正反两个问题：

- **正向问题：** 「如果外卖App有"实时骑手位置"功能，你的感受是？」
- **反向问题：** 「如果外卖App没有"实时骑手位置"功能，你的感受是？」

每个问题的回答选项：
```
A. 非常喜欢（这太棒了）
B. 理所应当（本来就该有）
C. 无所谓（有没有都行）
D. 勉强接受（不太喜欢但能忍）
E. 非常不喜欢（绝对不能接受）
```

**第三步：收集回答（20分钟）**

找 3-5 位朋友，让他们分别回答全部 10 个功能的正反问题。

**第四步：用分类矩阵判定类别（20分钟）**

根据正反答案的组合，判定功能类别：

| | 没有时：喜欢 | 没有时：应该有 | 没有时：无所谓 | 没有时：忍受 | 没有时：不喜欢 |
|---|---|---|---|---|---|
| **有时：喜欢** | 可疑 | 兴奋型 | 兴奋型 | 兴奋型 | 期望型 |
| **有时：应该有** | 反向 | 无差异 | 无差异 | 无差异 | 必备型 |
| **有时：无所谓** | 反向 | 无差异 | 无差异 | 无差异 | 必备型 |
| **有时：忍受** | 反向 | 无差异 | 无差异 | 无差异 | 必备型 |
| **有时：不喜欢** | 反向 | 反向 | 反向 | 反向 | 可疑 |

- **必备型（Must-be）：** 没有会不满，有了觉得理所当然
- **期望型（Performance）：** 越多越满意，越少越不满
- **兴奋型（Attractive）：** 没有不会不满，有了会惊喜
- **无差异（Indifferent）：** 有没有都不影响满意度

**第五步：绘制分类结果图 & 反思（15分钟）**

将 10 个功能标注到 Kano 坐标图上，回答：
1. 哪些功能是「必须做但不用花哨」的必备型？
2. 哪些功能能产生「哇」效果的兴奋型？
3. 如果预算只够做 4 个功能，你怎么选？

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 问卷设计清晰，分类准确，优先级选择有策略思考 |
| 良好 | 分类基本正确，但反思缺乏深度 |
| 需改进 | 分类错误较多，无法区分必备型和兴奋型 |

#### 预期收获

- 理解「有和没有」带来的不对称感受
- 学会区分「做了不加分但不做扣分」的基础功能
- 掌握用真实用户反馈代替猜测的方法

### 练习 3：用户故事写作工坊

**中文名称：** 「用户心声翻译机」

**时间估计：** 45 分钟

**格式说明：** 书面练习，将模糊需求转化为标准用户故事格式。

#### 具体步骤

**第一步：学习标准格式（5分钟）**

用户故事的标准模板：

```
作为一个 [用户角色]，
我想要 [实现某个目标]，
以便 [获得某种价值]。
```

好的用户故事满足 INVEST 原则：
- **I**ndependent（独立的）
- **N**egotiable（可协商的）
- **V**aluable（有价值的）
- **E**stimable（可估算的）
- **S**mall（足够小的）
- **T**estable（可测试的）

**第二步：从坏到好的改写练习（20分钟）**

将以下 5 个模糊需求改写为标准用户故事：

| 模糊需求 | 改写目标 |
|----------|----------|
| 「加个搜索功能」 | 写出谁在搜、搜什么、为什么搜 |
| 「页面太慢了」 | 写出哪个页面、什么场景、影响是什么 |
| 「要支持暗黑模式」 | 写出谁需要、什么时候需要、解决什么问题 |
| 「加个分享按钮」 | 写出分享给谁、分享什么内容、目的是什么 |
| 「需要数据导出」 | 写出谁要导出、什么格式、用来做什么 |

**改写示例：**

模糊需求：「加个搜索功能」

改写后：
```
作为一个拥有 200+ 笔记的重度用户，
我想要通过关键词搜索过往笔记内容，
以便在开会时快速找到相关记录而不用手动翻找。

验收标准：
- 支持标题和正文全文搜索
- 搜索结果在 2 秒内返回
- 搜索词在结果中高亮显示
- 无结果时显示友好提示
```

**第三步：原创用户故事撰写（15分钟）**

为你最熟悉的一个产品，撰写 3 个原创用户故事，每个都需包含：
- 用户角色（具体，不要用「用户」）
- 目标（可操作的行为）
- 价值（解决真实痛点）
- 验收标准（3-5 条，可测试）

**第四步：自我检查（5分钟）**

用 INVEST 原则逐条检查你写的每个用户故事。

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 用户角色具体、目标清晰、价值真实、验收标准可测试 |
| 良好 | 格式正确但角色或价值描述较泛 |
| 需改进 | 仍然是功能描述而非用户故事 |

#### 预期收获

- 从「功能导向」转变为「用户导向」思维
- 学会用验收标准明确需求边界
- 能清晰向 AI 工具表达你想要构建的东西

### 练习 4：Jobs-to-be-Done 深度重构

**中文名称：** 「需求真相挖掘机」

**时间估计：** 60 分钟

**格式说明：** 思维练习 + 书面分析，解构日常产品的真实「工作」。

#### 具体步骤

**第一步：理解经典案例（10分钟）**

阅读并理解奶昔案例：

快餐店发现奶昔销量集中在早上 8:30 前，买家多是独自开车的通勤者。他们「雇佣」奶昔的真正工作不是「我想喝奶昔」，而是「我需要一个东西让无聊的通勤不那么无聊，同时撑到午饭」。竞争对手不是其他奶昔品牌，而是香蕉、百吉饼和甜甜圈。

**第二步：JTBD 公式练习（20分钟）**

JTBD 陈述公式：
```
当我 [处于某个情境] 时，
我想要 [完成某个动机]，
这样我就能 [达到预期结果]。
```

为以下 6 个产品撰写 JTBD 陈述（每个产品写功能、情感、社交三个维度）：

| 产品 | 功能维度 | 情感维度 | 社交维度 |
|------|----------|----------|----------|
| 滴滴出行 | 「当我赶时间时，我想要快速叫到车...」 | 「让我不用焦虑等车...」 | 「让我显得有品位/效率高...」 |
| 小红书 | ? | ? | ? |
| 钉钉/飞书 | ? | ? | ? |
| 拼多多 | ? | ? | ? |
| Keep健身 | ? | ? | ? |
| 得到/喜马拉雅 | ? | ? | ? |

**第三步：竞品替代分析（15分钟）**

对上面每个产品回答：「如果这个产品明天消失，你会用什么替代？」

列出至少 3 个替代方案，包括：
- 直接竞品（同类App）
- 间接竞品（不同形式但解决同一工作）
- 非数字替代（完全离线的解决方案）

例如：Keep 消失后的替代方案
1. 直接竞品：薄荷健身、Nike Training
2. 间接竞品：B站健身视频、朋友圈打卡
3. 非数字替代：去健身房请私教

**第四步：洞察提炼（15分钟）**

从你的分析中提炼 3 个洞察：
1. 哪个产品的「真实工作」和它的广告宣传差距最大？
2. 哪个产品的竞争对手最出人意料？
3. 如果你要创业，这些洞察指向什么机会？

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 三个维度区分清晰，替代分析有意外发现，洞察深刻 |
| 良好 | JTBD 陈述正确但维度区分不够明确 |
| 需改进 | 将功能描述误认为 JTBD |

#### 预期收获

- 跳出「功能思维」，理解用户真正要完成的工作
- 重新定义竞争格局（真正的竞争对手可能来自完全不同的领域）
- 为产品创新找到新角度

### 练习 5：产品-市场契合度评估

**中文名称：** 「PMF 体检报告」

**时间估计：** 60 分钟

**格式说明：** 分析练习，对一个真实产品进行 PMF 多维评估。

#### 具体步骤

**第一步：选择评估对象（5分钟）**

选择一个你认为「找到了 PMF」的产品和一个你认为「还没找到 PMF」的产品。

**第二步：Sean Ellis 测试模拟（15分钟）**

Sean Ellis 的核心问题：「如果明天不能再使用这个产品，你会感到？」

- 非常失望
- 有点失望
- 不太失望
- 不适用（已经不用了）

如果超过 40% 的用户选择「非常失望」，产品已达到 PMF。

找 5-10 个朋友（或在社交媒体上发起投票），对你选择的两个产品分别进行这个测试。记录结果并计算百分比。

**第三步：PMF 信号检查表（20分钟）**

对每个产品，逐条检查以下信号：

```
增长信号：
[ ] 用户自发推荐给他人（有口碑效应）
[ ] 不需要大量广告就能获得新用户
[ ] 用户使用频率在增加
[ ] 用户留存率稳定或上升

产品信号：
[ ] 核心功能清晰（一句话说清楚产品是什么）
[ ] 用户知道在什么场景下使用它
[ ] 竞品用户愿意切换过来
[ ] 用户愿意付费或有明确的付费意愿

市场信号：
[ ] 目标市场足够大
[ ] 用户痛点真实且频繁发生
[ ] 现有解决方案明显不足
[ ] 时机合适（技术、政策、用户习惯已就绪）
```

**第四步：撰写评估报告（20分钟）**

用以下模板撰写 300-500 字的评估报告：

```
产品名称：______
PMF 判定：已达到 / 接近 / 未达到

核心证据：
1. ______
2. ______
3. ______

最大风险：______

建议下一步行动：______
```

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 两个产品对比鲜明，证据具体，判定有说服力 |
| 良好 | 判定合理但证据偏主观 |
| 需改进 | 无法区分 PMF 信号和产品功能好坏 |

#### 预期收获

- 理解 PMF 不是「产品好不好」而是「市场要不要」
- 学会用结构化方式评估创业想法
- 认识到 PMF 是持续验证过程而非一次性判断

### 练习 6：竞品分析全景图

**中文名称：** 「竞争格局扫描仪」

**时间估计：** 90 分钟

**格式说明：** 综合分析练习，完成一份完整竞品分析文档。

#### 具体步骤

**第一步：定义分析范围（10分钟）**

选择一个你感兴趣的产品领域（如笔记工具、短视频、在线教育），列出：
- 3 个直接竞品（同类产品）
- 2 个间接竞品（不同方式解决同一问题）

**第二步：信息收集矩阵（30分钟）**

为每个竞品填写以下矩阵：

| 维度 | 竞品A | 竞品B | 竞品C | 竞品D | 竞品E |
|------|-------|-------|-------|-------|-------|
| 核心定位（一句话） | | | | | |
| 目标用户 | | | | | |
| 定价策略 | | | | | |
| 核心功能（前3个） | | | | | |
| 独特卖点 | | | | | |
| 最大弱点 | | | | | |
| 用户评价倾向 | | | | | |

**第三步：功能对比矩阵（20分钟）**

列出该领域 15-20 个关键功能，对每个竞品标注：

- 「强」= 该功能是行业最佳
- 「有」= 有此功能，表现一般
- 「弱」= 有此功能但体验差
- 「无」= 没有此功能

**第四步：市场定位地图（15分钟）**

画一个二维坐标图，选择两个最重要的维度作为坐标轴（例如：价格 vs 功能丰富度，或专业程度 vs 易用性），将 5 个竞品标注在图上。

**第五步：SWOT 总结 + 机会发现（15分钟）**

选择你最看好的一个产品，完成 SWOT 分析：

```
优势 (S)：         劣势 (W)：
1.                 1.
2.                 2.

机会 (O)：         威胁 (T)：
1.                 1.
2.                 2.
```

最后回答：如果你是这个产品的 PM，你会做什么来拉开与竞品的差距？

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 信息准确全面，定位图有洞察，SWOT分析深刻 |
| 良好 | 信息基本完整但分析深度不足 |
| 需改进 | 只是罗列信息，缺乏分析和判断 |

#### 预期收获

- 掌握系统性分析竞争格局的方法
- 学会从竞品差异中发现产品机会
- 建立「知己知彼」的产品思维

### 练习 7：用户访谈模拟训练

**中文名称：** 「用户心声采集器」

**时间估计：** 75 分钟

**格式说明：** 角色扮演练习，与一位朋友进行模拟用户访谈。

#### 具体步骤

**第一步：准备访谈大纲（15分钟）**

选择一个访谈主题（如「了解人们如何管理个人财务」），按以下结构准备问题：

```
开场（建立信任，2-3分钟）：
- 简单介绍自己和访谈目的
- 强调没有对错之分
- 询问是否可以记录

背景了解（5分钟）：
Q1: 能描述一下你平时是怎么管理个人财务的吗？
Q2: 你一般用什么工具来记账或理财？

行为挖掘（10分钟）：
Q3: 上一次你为一笔开支感到纠结是什么时候？能描述一下当时的情况吗？
Q4: 你最近一次查看自己的财务状况是什么时候？是什么触发了这个行为？

痛点探索（10分钟）：
Q5: 在管理财务的过程中，最让你头疼的是什么？
Q6: 你有没有尝试过解决这个问题？结果怎么样？

情感挖掘（5分钟）：
Q7: 当你的财务状况在掌控中时，你是什么感受？
Q8: 如果有一个魔法工具能帮你解决一个财务问题，你希望它做什么？

收尾（2分钟）：
Q9: 还有什么我没问到但你觉得重要的？
```

**第二步：关键访谈技巧备忘（5分钟）**

在访谈前复习这些规则：

```
必须做的：
- 多用「为什么」「能详细说说吗」「然后呢」追问
- 让对方讲故事（「能描述一个具体的例子吗」）
- 记录用户的原话，不要翻译
- 沉默也是工具——对方停顿时不要急着填充

绝对不做的：
- 不要问引导性问题（「你是不是觉得XXX很好用？」）
- 不要推销你的想法（「如果有个App能XXX，你会用吗？」）
- 不要打断对方
- 不要在对方面前做价值判断（「那个方法不太好」）
```

**第三步：进行模拟访谈（30分钟）**

找一位朋友，按照你准备的大纲进行 20-25 分钟的访谈。全程录音或做笔记。

**第四步：访谈分析（25分钟）**

访谈结束后，整理以下内容：

```
关键发现：
1. 用户最大的痛点是什么？（用原话引用）
2. 用户目前的解决方案是什么？为什么不够好？
3. 有什么出乎你意料的发现？

洞察提炼：
- 功能需求：______
- 情感需求：______
- 潜在需求（用户没说但你推断出的）：______

自我反思：
- 哪个问题问得最好？引出了什么信息？
- 哪个地方你犯了「引导性提问」的错误？
- 下次你会怎么改进问题列表？
```

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 问题开放且深入，没有引导性提问，分析有洞察 |
| 良好 | 基本遵循技巧但追问不够深入 |
| 需改进 | 频繁使用引导性或封闭式问题 |

#### 预期收获

- 掌握非引导性提问的技巧
- 学会从用户行为而非陈述中提取需求
- 建立与用户共情的能力

### 练习 8：产品指标仪表盘设计

**中文名称：** 「数据洞察罗盘」

**时间估计：** 60 分钟

**格式说明：** 思维练习 + 图表绘制，为一个虚拟产品设计核心指标体系。

#### 具体步骤

**第一步：选择产品场景（5分钟）**

选择以下之一（或自选）：
- A. 一个新上线的读书App
- B. 一个社区团购小程序
- C. 一个在线学习平台

**第二步：定义北极星指标（10分钟）**

北极星指标 = 最能代表产品核心价值的单一指标

```
好的北极星指标满足：
- 反映用户获得的核心价值
- 可以被团队影响
- 领先指标（预示未来增长）

示例：
- Airbnb：预订间夜数
- Spotify：每周收听时长
- Slack：发送的消息数
- 读书App：每周完成阅读章节数
```

为你选择的产品定义一个北极星指标，并解释为什么选它。

**第三步：设计指标层级（20分钟）**

围绕北极星指标，设计三层指标体系：

```
第一层：北极星指标
  └── 你选择的核心指标

第二层：驱动指标（影响北极星的关键因素）
  ├── 获客指标（新用户如何来）
  │   ├── 日新增用户数
  │   ├── 获客渠道分布
  │   └── 获客成本 (CAC)
  ├── 激活指标（新用户如何变活跃）
  │   ├── 注册转化率
  │   ├── 首次关键行为完成率
  │   └── 激活时间（从注册到首次关键行为的时长）
  ├── 留存指标（用户是否持续使用）
  │   ├── 次日留存率
  │   ├── 7日留存率
  │   ├── 30日留存率
  │   └── DAU/MAU 比率
  ├── 收入指标（如何变现）
  │   ├── 付费转化率
  │   ├── ARPU（每用户平均收入）
  │   └── LTV（用户生命周期价值）
  └── 推荐指标（用户是否传播）
      ├── 推荐率
      └── 病毒系数 (K-factor)

第三层：观察指标（辅助判断的细分数据）
  └── 按渠道/平台/用户群的细分数据
```

**第四步：留存曲线分析练习（15分钟）**

以下是一个虚拟产品的留存数据，绘制留存曲线并回答问题：

```
注册首日（Day 0）：1000 用户
Day 1：420 用户（42%留存）
Day 3：280 用户（28%）
Day 7：200 用户（20%）
Day 14：160 用户（16%）
Day 30：140 用户（14%）
Day 60：130 用户（13%）
Day 90：125 用户（12.5%）
```

问题：
1. 这条留存曲线健康吗？为什么？
2. 最大的流失发生在哪个时间段？意味着什么？
3. 如果你是 PM，你会优先改善哪个阶段的留存？

**第五步：队列分析理解（10分钟）**

理解队列分析（Cohort Analysis）的概念：

```
将用户按注册时间分组，观察每组的行为变化：

        | 第1周 | 第2周 | 第3周 | 第4周 |
1月注册 | 100%  | 35%   | 22%   | 18%   |
2月注册 | 100%  | 38%   | 25%   | 20%   |
3月注册 | 100%  | 42%   | 30%   | 25%   |
```

问题：这个趋势说明什么？（提示：后注册的用户留存更好，产品可能在变好）

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 北极星指标选择精准，指标体系有层次，留存分析有洞察 |
| 良好 | 指标选择合理但体系不够完整 |
| 需改进 | 无法区分虚荣指标和可操作指标 |

#### 预期收获

- 理解指标之间的因果关系
- 学会从留存曲线中读出产品健康度
- 建立数据驱动的产品决策思维

### 练习 11：Nielsen 十大可用性原则审计

**中文名称：** 「UX 体检全科医生」

**时间估计：** 75 分钟

**格式说明：** 系统化审计练习，逐条用 Nielsen 十大原则评估一个真实产品。

#### 具体步骤

**第一步：理解十大原则（15分钟）**

逐条阅读并用自己的话重述每条原则：

```
1. 系统状态可见性
   系统应始终通过合理的反馈，在合理的时间内让用户知道正在发生什么。
   日常例子：电梯的楼层显示、手机信号格数

2. 系统与现实世界的匹配
   系统应使用用户熟悉的语言、概念和逻辑，而非技术术语。
   日常例子：回收站图标、购物车隐喻

3. 用户控制与自由
   用户经常误操作，需要明确的「紧急出口」来离开非预期状态。
   日常例子：撤销功能 (Ctrl+Z)、Gmail的「撤回发送」

4. 一致性与标准
   用户不应猜测不同的词语、场景或操作是否意味着同一件事。
   日常例子：所有页面的导航栏位置一致

5. 错误预防
   比好的错误提示更好的是——一开始就防止错误发生。
   日常例子：删除前的确认弹窗、表单的格式提示

6. 识别而非回忆
   通过让对象、操作和选项可见来减轻用户的记忆负担。
   日常例子：搜索历史、最近使用的文件

7. 使用的灵活性与效率
   快捷方式对新手不可见，但能加速专家用户的操作。
   日常例子：键盘快捷键、手势操作

8. 美观与简约设计
   界面不应包含不相关或很少需要的信息。每增加一条信息都在与核心信息竞争。
   日常例子：Google首页的极简设计

9. 帮助用户识别、诊断和恢复错误
   错误信息应用通俗语言表达，准确指出问题，并建议解决方案。
   日常例子：「密码需要至少8个字符，包含数字和字母」

10. 帮助与文档
    系统最好不需要文档就能使用，但必要时应提供易搜索的帮助信息。
    日常例子：产品内的帮助中心、FAQ
```

**第二步：选择审计对象并执行（40分钟）**

选择一个你每天使用的 App，逐条审计。用以下打分表：

```
产品名称：______
审计日期：______

┌──────────────────────────────────────────────────┐
│ 原则1：系统状态可见性                              │
│                                                    │
│ 做得好的例子：                                      │
│ __________________________________________________ │
│                                                    │
│ 做得差的例子：                                      │
│ __________________________________________________ │
│                                                    │
│ 改进建议：                                          │
│ __________________________________________________ │
│                                                    │
│ 严重程度：□无问题  □轻微  □中等  □严重  □灾难性    │
│ 评分（1-5分）：___                                  │
└──────────────────────────────────────────────────┘

（对全部10条原则重复以上模板）
```

严重程度评定标准：
- 0-无问题：完全符合原则
- 1-轻微：影响美观但不影响使用
- 2-中等：用户需要额外思考但能完成任务
- 3-严重：部分用户会放弃任务
- 4-灾难性：几乎所有用户会受阻

**第三步：撰写审计报告摘要（20分钟）**

```
审计总结：
- 总分：___/50
- 最强的3条原则：______
- 最弱的3条原则：______
- 优先改进建议（按严重程度排序）：
  1. ______
  2. ______
  3. ______
```

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 每条原则都有具体例子和截图，严重程度判断合理，改进建议可操作 |
| 良好 | 大部分原则有例子但部分缺少改进建议 |
| 需改进 | 例子模糊或与原则不对应 |

#### 预期收获

- 建立系统化的 UX 评估能力（不再是「感觉不太对」）
- 学会用专业术语描述设计问题
- 能向设计师或 AI 工具精确表达需要改进的地方

### 练习 21：暗黑模式猎人

**中文名称：** 「操纵设计曝光台」

**时间估计：** 60 分钟

**格式说明：** 侦察收集练习，在真实网站中寻找并分类暗黑模式。

#### 具体步骤

**第一步：学习暗黑模式分类（10分钟）**

```
1. 诱饵和替换（Bait and Switch）
   用户以为在做A操作，实际触发了B操作
   例：点击「关闭」按钮却打开了广告

2. 伪装广告（Disguised Ads）
   广告伪装成内容或导航元素
   例：下载页面中假的「下载」按钮

3. 强迫续费（Forced Continuity）
   免费试用要求绑定信用卡，到期自动扣费
   例：7天免费试用，不主动取消则按年扣费

4. 偷偷加料（Sneak into Basket）
   结账时自动添加额外商品或服务
   例：机票预订时默认勾选保险

5. 蟑螂旅馆（Roach Motel）
   注册容易取消难
   例：一键注册 vs 需要打电话才能注销

6. 隐私侵蚀（Privacy Zuckering）
   诱导用户分享更多隐私数据
   例：大大的「接受所有Cookie」vs 小小的「自定义设置」

7. 确认羞辱（Confirmshaming）
   用羞辱性语言让用户不好意思拒绝
   例：「不，我不想省钱」作为拒绝按钮

8. 紧迫感制造（Urgency）
   虚假的时间压力或库存紧张
   例：「仅剩3件」「还有2人在看」

9. 误导性问题（Trick Questions）
   用双重否定或复杂措辞混淆用户
   例：「取消勾选以不接收非促销类邮件」

10. 视觉干扰（Misdirection）
    用视觉设计将注意力引向对公司有利的选项
    例：「升级」按钮是彩色大按钮，「保持原计划」是灰色小字
```

```
- 字间距（letter-spacing）
- 列表项之间的间距
- 图标与文字之间的间距
- 按钮内部的padding
```

### 练习 24：栅格系统理解训练

**中文名称：** 「隐形骨架透视镜」

#### 具体步骤

**第一步：理解栅格三要素（10分钟）**

```
栅格由三个部分组成：

1. 列（Columns）
   - 内容放置的垂直区域
   - 常见配置：12列（桌面端）、4列（移动端）、8列（平板端）
   - 为什么是12？因为12能被2、3、4、6整除，布局灵活

2. 间距（Gutters）
   - 列与列之间的空隙
   - 常见值：16px、20px、24px
   - 保证内容不会粘在一起

3. 边距（Margins）
   - 内容区域与屏幕边缘的距离
   - 常见值：桌面端 120-180px，移动端 16-24px
   - 确保内容不会贴着屏幕边缘

常见布局模式：
全宽（12列） → 大图、横幅
两等分（6+6列） → 左右对称布局
主次分（8+4列 或 9+3列） → 内容区 + 侧边栏
三等分（4+4+4列） → 卡片列表
四等分（3+3+3+3列） → 商品列表
```

### 练习 25：图标一致性评估

**中文名称：** 「图标风格质检员」

**时间估计：** 30 分钟

**格式说明：** 视觉评估练习，检查同一产品内图标的风格统一性。

#### 具体步骤

**第一步：理解图标风格维度（5分钟）**

```
评估图标一致性的 6 个维度：

1. 线条粗细（Stroke Width）
   所有图标的线条粗细应该一致（如都是2px）

2. 填充 vs 线条（Filled vs Outlined）
   同一组图标应统一为填充风格或线条风格
   常见做法：未选中=线条，选中=填充

3. 圆角 vs 直角（Rounded vs Sharp）
   所有图标的角应该一致（都圆角或都直角）

4. 尺寸（Size）
   同级别的图标大小应该一致（如导航图标都是24x24px）

5. 视觉重量（Visual Weight）
   不同图标在同一尺寸下看起来应该「一样重」
   （圆形图标天然比方形看起来小，需要适当放大）

6. 风格统一（Style Consistency）
   不要混用不同图标库的图标（如一个来自Material、一个来自SF Symbols）
```


### 练习 29：设计系统意识建立

**中文名称：** 「设计乐高积木认知」

**时间估计：** 45 分钟

**格式说明：** 拆解分析练习，理解设计系统如何保证产品的一致性和效率。

#### 具体步骤

**第一步：理解设计系统的概念（10分钟）**

```
设计系统 = 一套可复用的设计规则和组件

把它想象成「乐高积木」：
- 你不需要每次都从零开始造一个新的积木块
- 而是用标准化的积木块自由组合出不同的作品
- 每个积木块的尺寸、颜色、连接方式都是标准化的

设计系统通常包含：
├── 基础层（Design Tokens）
│   ├── 颜色体系（Primary, Secondary, Neutral, Error...）
│   ├── 字体体系（字号、行高、字重的完整列表）
│   ├── 间距体系（4px, 8px, 12px, 16px, 24px, 32px, 48px...）
│   ├── 圆角体系（0px, 4px, 8px, 12px, 999px...）
│   └── 阴影体系（sm, md, lg...）
│
├── 组件层（Components）
│   ├── 按钮（Primary, Secondary, Ghost, Danger...）
│   ├── 输入框（Text, Number, Password, Search...）
│   ├── 卡片（Content Card, Metric Card, User Card...）
│   ├── 导航（Navbar, Sidebar, Tabs, Breadcrumb...）
│   └── 反馈（Toast, Modal, Alert, Tooltip...）
│
└── 模式层（Patterns）
    ├── 表单模式
    ├── 列表模式
    ├── 详情页模式
    └── 仪表盘模式
```

**第二步：拆解一个真实的设计系统（20分钟）**

选择一个公开的设计系统进行研究（推荐：Ant Design / Material Design / Shadcn UI）。

```
设计系统名称：______
网址：______

颜色体系分析：
- 主色有几个变体（从浅到深）？___个
- 系统灰色有几级？___级
- 功能色（成功/警告/错误/信息）各是什么颜色？
  成功：#______ 警告：#______ 错误：#______ 信息：#______

字体体系分析：
- 使用的字体家族：______
- 最大标题字号：___px
- 正文字号：___px
- 最小字号：___px
- 字号之间的递进规律是什么？（等比？等差？）______

间距体系分析：
- 最小间距：___px
- 间距递进规律：______（如：4的倍数）
- 组件内部间距与组件之间间距的关系：______

按钮组件分析：
- 提供了几种按钮类型？______
- 每种按钮有几种状态（默认/悬停/按下/禁用）？___种
- 按钮的尺寸有几种（大/中/小）？___种
```

**第三步：设计系统感知测试（15分钟）**

不看设计系统文档，仅通过使用一个 App（推荐使用对应的产品，如 Ant Design → 蚂蚁相关产品），尝试反推它的设计规则：

```
我观察到的规则：

颜色规则：
- 主色好像是______
- 链接颜色是______
- 错误提示颜色是______

间距规则：
- 卡片之间的间距大约是___px
- 卡片内部的内边距大约是___px
- 列表项之间的间距大约是___px

组件规则：
- 所有按钮的圆角大约是___px
- 所有卡片的圆角大约是___px
- 两者是否一致？□是 □否

然后查看官方文档验证你的推测：
- 猜对了几条？___/总数
- 最出乎意料的发现：______
```

#### 评估标准

| 等级 | 标准 |
|------|------|
| 优秀 | 拆解完整，反推准确率高，理解设计系统的价值 |
| 良好 | 拆解了大部分但反推准确率一般 |
| 需改进 | 无法识别设计系统的基础层要素 |

#### 预期收获

- 理解为什么好的产品「看起来就很统一」
- 认识到设计系统是效率和一致性的保障
- 在 vibe coding 时，知道应该让 AI 使用哪个设计系统（如 Shadcn）
